gokaf — Build Your Own Kafka Broker in Go

Master plan: learn Go syntax and deep Kafka internals by re-implementing a Kafka-compatible broker from scratch — protocol, storage engine, consumer groups, replication, and a simplified controller.

Context & decisions

BackgroundComing from .NET / C#. Go syntax notes are called out per phase where the mental model differs meaningfully (error handling, interfaces, concurrency).
ScopeFull scope, including multi-broker replication and a simplified controller (Track D + E), not just a single-broker implementation.
Client libraryNone for the broker itself — this project is the broker. A hand-written test client is built in parallel (cmd/testclient) instead of using a library like kafka-go or franz-go.
ReferenceArchitecture inspiration only from KhaiHust/kaf-go. Implementation is written from the official Kafka Protocol Guide and KIP-98 (RecordBatch v2), not copied from kaf-go.
VerificationPrimary: hand-written test client. Recommended (not required) sanity check with a real client such as kcat at major milestones (end of Phase 8, end of Phase 15) to catch protocol drift early.

Honest scope warning

Track D and Track E are substantially harder than Track A–C, both in code volume and in how hard distributed-systems bugs are to reproduce. Phase 23 explicitly does not implement real Raft consensus — it borrows the idea, not the safety guarantees. If real Raft is wanted later, treat it as its own project (start by reading etcd/raft), not bolted onto this one.

Repo structure

  cmd/
    broker/         # main.go — runs one broker node
    testclient/      # hand-written test client (no kafka-go/franz-go)

    network/             # TCP server, connection handling, request dispatch

    topic/                   # topic/partition metadata registry
    group/                     # consumer group coordinator state machine
    producer/                    # PID manager, sequence dedup
    replication/                   # ReplicaFetcher, ISR (Track D)
    cluster/                         # controller election, metadata log (Track E)
  docs/
    master-plan.html                  # this file
    protocol-notes.md                   # your own notes while reading the spec
  go.mod

Testing approach

Every phase's acceptance criteria is verified by a hand-written Go test client talking raw bytes over TCP to the broker — no client library. This is deliberate: writing both sides forces confronting the wire format directly, at the cost of a real risk that a bug in the client silently "matches" a bug in the server. Mitigate this by cross-checking with kcat against a real Kafka broker whenever a response looks suspiciously convenient.

Track A — Protocol & Single-Broker Core

Phases 0–10

The foundation: wire protocol, storage engine, Produce/Fetch. Everything else builds on this.

Phase 0 TCP skeleton + spec reading

Input
Go 1.22+. Read the Kafka Protocol Guide sections "Preliminaries" and "Wire Format" before writing code.
Output
A net.Listener on TCP port 9092. Each connection reads the 4-byte size prefix, reads that many bytes, and logs a hex dump. No parsing yet.
Acceptance criteria
nc -v localhost 9092 sends arbitrary bytes → broker logs the correct byte count and hex dump, no crash.
Go concepts
  • net.Listener / net.Conn
  • One goroutine per connection
  • io.ReadFull, encoding/hex
Kafka concepts
  • Size-prefixed framing underlies the entire protocol

Phase 1 Binary primitive encode/decode

Input
TCP skeleton from Phase 0.
Output
internal/protocol/types.go with read/write helpers for INT8/16/32/64, UNSIGNED_VARINT, VARINT (zigzag), STRING, COMPACT_STRING, NULLABLE_STRING, ARRAY, COMPACT_ARRAY, and TAG_BUFFER (minimal: a single 0x00 byte means "no tagged fields").
Acceptance criteria
Unit tests round-trip encode→decode for every type and match the worked byte examples in the protocol guide.
Go concepts
  • bytes.Buffer, encoding/binary.BigEndian
  • Custom Decoder/Encoder structs wrapping io.Reader/io.Writer

Phase 2 Request header parsing + dispatch skeleton

Input
Primitive encode/decode from Phase 1.
Output
Parse RequestHeader (api_key, api_version, correlation_id, client_id). A dispatch table (map[int16]HandlerFunc) routes by api_key; unimplemented handlers return a well-formed "unsupported API" error instead of crashing.
Acceptance criteria
Test client sends a request with an unknown api_key → broker replies with a correct ResponseHeader{correlation_id} + error code, no crash.
Go concepts
  • Dispatch table instead of a long switch statement — functions as values

Phase 3 ApiVersions (key=18)

Output
Handler returns the list of API keys the broker supports with min/max version — the handshake every real client sends first.
Acceptance criteria
Test client sends ApiVersionsRequest → correctly parses the response.
Kafka concepts
  • Why ApiVersions is always the first request — the client needs to know which versions the server supports before sending anything else

Phase 4 Topic registry + Metadata (key=3)

Output
internal/topic/registry.go — a map of topics guarded by sync.RWMutex; Metadata handler returns cluster info (single broker = itself) plus the topic list (hardcode 1–2 test topics for now).
Acceptance criteria
MetadataRequest(topics=nil) returns the correct topic/partition/leader list.
Go concepts
  • sync.RWMutex for read-heavy shared state

Phase 5 CreateTopics / DeleteTopics (key=19/20)

Output
Handlers add/remove topics from the registry at runtime (no disk persistence yet).
Acceptance criteria
Create topic "orders" with 3 partitions → Metadata reflects it correctly.
Go concepts
  • Input validation using Go's return-error pattern

Phase 6 Storage engine — segmented log

Output
internal/storage/log.go. Each partition is a directory containing append-only segment files. Append(records []byte) (baseOffset int64, err error) and Read(offset int64, maxBytes int) ([]byte, error) (linear scan, no index yet).
Acceptance criteria
Append 1000 fake records, read back correctly at arbitrary offsets. Restart the process (close/reopen the file) → data survives.
Go concepts
  • os.OpenFile(O_APPEND), bufio.Writer, io.Seeker
Kafka concepts
  • Why append-only sequential I/O beats random-write databases for this workload

Phase 7 Produce API (key=0)

Output
Parse a v2 RecordBatch (batch header + varint-encoded records), validate the CRC32C, call Log.Append, return the assigned offset.
Acceptance criteria
Test client hand-encodes a valid RecordBatch → sends Produce → receives the correct base offset. Verify the on-disk bytes match the worked examples in KIP-98.
Kafka concepts
  • RecordBatch structure; why compression, transactions, and idempotency all live at the batch-header level

Phase 8 Fetch API (key=1)

Output
Parse a FetchRequest → Log.Read → re-encode as a RecordBatch → return the response.
Acceptance criteria
Produce 10 messages → Fetch from offset 0 → receive all 10, correct order, correct content.
Known limitation: no long-poll (fetch.max.wait.ms) yet — returns empty immediately if there's no data. Upgrade later if desired.

Phase 9 Offset index + segment rolling

Output
A sparse .index file alongside each .log file (offset → byte position, one entry per N bytes), plus segment rolling once a size limit is reached.
Acceptance criteria
Generate enough data to produce 2+ segments; Fetch from a later segment returns correct data. Benchmark shows lookup is meaningfully faster than the Phase 6 linear scan.
Go concepts
  • sort.Search (binary search), go test -bench

Phase 10 ListOffsets (key=2) + full multi-partition support

Output
Handler returns earliest/latest offset for a partition. Produce/Fetch work correctly when a topic has multiple partitions (each with its own Log).
Acceptance criteria
Test client uses ListOffsets to compute where to start fetching from, like a real consumer would, verified on a multi-partition topic.

Track B — Consumer Group

Phases 11–15

Group coordination protocol: how consumers discover each other, split partitions, and track progress.

Phase 11 FindCoordinator (key=10)

Output
Single-broker setup, so the coordinator is always this broker. Returns broker info as coordinator for a given group_id.
Acceptance criteria
Correct broker info returned for any group_id.
Kafka concepts
  • What a coordinator is — the one broker responsible for a given consumer group's state

Phase 12 JoinGroup / SyncGroup (key=11/14) + state machine

Output
internal/group/coordinator.go: state machine (Empty → PreparingRebalance → CompletingRebalance → Stable). JoinGroup registers members and picks one as leader; SyncGroup distributes the leader's partition assignment to every member.
Acceptance criteria
2–3 simulated consumers in the same group → one is elected leader → SyncGroup assigns partitions with no overlap and no gaps.
Go concepts
  • State machine via type GroupState int + iota — Go has no sealed-class equivalent, unlike Kotlin

Phase 13 Heartbeat / LeaveGroup (key=12/13) + session timeout

Output
Heartbeat resets a per-member timer. No heartbeat within session.timeout.ms → coordinator evicts the member and triggers a rebalance.
Acceptance criteria
Simulate one member crashing (stop sending heartbeats) → after the timeout, the coordinator rebalances and the remaining members pick up its partitions.
Go concepts
  • time.AfterFunc per member — careful to Stop() the timer on departure to avoid goroutine leaks

Phase 14 OffsetCommit / OffsetFetch (key=8/9) + internal offsets log

Output
Use the Phase 6 storage engine to create an internal __consumer_offsets topic. Commit writes a record (group_id + topic + partition → offset); OffsetFetch reads the latest state.
Acceptance criteria
Commit an offset, fully restart the broker (kill + start), OffsetFetch still returns the committed offset — persistence across restarts.
Kafka concepts
  • This is exactly how real Kafka stores offsets — an internal topic, not a separate database table

Phase 15 Real Range / RoundRobin assignors

Output
Replace the placeholder assignment from Phase 12 with two real strategies (Range, RoundRobin), selectable via JoinGroup protocol metadata.
Acceptance criteria
Pure unit test of the assignor (no network): given N partitions and M members, output matches the worked examples in the Kafka docs.

Track C — Producer Reliability

Phase 16

Idempotency at the partition level — the real mechanism behind Kafka's exactly-once-per-partition guarantee.

Phase 16 InitProducerId (key=22) + idempotent producer

Output
A PID manager assigns producer_id/producer_epoch per producer session. The broker tracks (producer_id, partition) → last_sequence to deduplicate — a repeated sequence number is silently dropped instead of re-appended.
Acceptance criteria
Test client deliberately resends the same batch twice (simulating a retry after a timeout) → the log contains exactly one copy, not two.
Kafka concepts
  • This is Kafka's real exactly-once-per-partition mechanism — no cross-partition transaction needed

Track D — Multi-Broker & Replication

Phases 17–21 · heavy

This is where the project stops being "one process" and starts being a distributed system.

Phase 17 Multi-broker bootstrap

Output
Run N broker instances (each with a broker_id and its own port), each aware of the others via static config (no service discovery yet). internal/cluster/brokerclient.go reuses the Track A protocol layer for inter-broker calls.
Acceptance criteria
3 brokers started together → each broker's Metadata response correctly lists all 3 (id, host, port).

Phase 18 Partition leader assignment across brokers

Output
CreateTopics distributes each partition's leader round-robin across brokers instead of always the local broker. Metadata reflects the real leader per partition.
Acceptance criteria
Create a topic with 6 partitions on a 3-broker cluster → leaders are evenly spread, 2 partitions per broker.

Phase 19 ReplicaFetcher — follower replication

Output
Any broker that isn't the leader for a partition runs a goroutine loop continuously fetching from the leader (via the existing Fetch API over BrokerClient) and appending to its own local log.
Acceptance criteria
Produce to the leader → within a few seconds, the follower brokers' log files have matching checksums.

Phase 20 ISR tracking + high-watermark + acks=all

Output
The leader tracks which followers are "caught up" (the in-sync replica set). Produce(acks=all) only returns once enough ISR members have fetched up to that offset (the high-watermark).
Acceptance criteria
Simulate a slow follower (delay its fetch loop) → Produce(acks=all) blocks/waits correctly, while Produce(acks=1) returns immediately without waiting.
Kafka concepts
  • This is the core of Kafka's durability guarantee — understanding this phase is understanding roughly 80% of "real Kafka"

Phase 21 Simple leader failover

Output
When the leader broker goes down (detected via inter-broker heartbeat), another broker in the ISR is promoted to leader. Metadata is updated for clients.
Acceptance criteria
Kill the leader process mid-Produce → the producer (following the NotLeaderForPartition error) switches to the new leader without losing any data that was already in the ISR at crash time.
Known limitation: this failover rule is simplified — "lowest broker ID still alive in the ISR wins" — not a real controller decision. Replaced by a real controller in Phase 22–23.

Track E — Controller / Cluster Metadata

Phases 22–23 · very advanced

A deliberately simplified take on what KRaft does for real. Treat these two phases as "learn the shape of the idea," not "ship a production consensus system."

Phase 22 Controller election

Output
One broker in the cluster is elected "controller" and becomes solely responsible for deciding partition leadership, replacing the ad-hoc rule from Phase 21. Simplified via a bully algorithm (highest broker ID still alive wins) instead of real Raft.
Acceptance criteria
Kill the current controller → the cluster elects exactly one new controller within a few seconds, with no split-brain, under the simplifying assumption that there's no network partition in the test.

Phase 23 KRaft-style metadata log (optional)

Output
The controller writes every cluster change (topic create/delete, leader changes, broker join/leave) to its own append-only log (reusing the Track A storage engine). Other brokers replay this log to stay in sync — this is the core idea behind KRaft.
Acceptance criteria
Restart the entire cluster → every broker replays the log and ends up with identical metadata.
Known limitation: this does not implement real Raft consensus and does not guarantee safety under an actual network partition or split-brain scenario. It teaches the idea, not a production-safe implementation. Real Raft belongs in its own project — start by reading etcd/raft, don't build it from scratch bolted onto this broker.
